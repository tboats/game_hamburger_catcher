# Output Format

Official template for the `/scan-sec` report. SKILL.md and workflows must adhere to this format. All user-facing text must use i18n keys (see [`i18n/vi.md`](i18n/vi.md) and [`i18n/en.md`](i18n/en.md)) — DO NOT hardcode translations directly.

## Structure Overview (v0.3+)

```
1. Header block (metadata)
2. Verdict line (PASS/WARN/FAIL + 1-line description)
3. Section: CRITICAL — Overview (compact table) → Verbose blocks per finding
4. Section: HIGH — Overview (compact table) → Verbose blocks per finding
5. Section: MEDIUM (compact table, 1 row per finding)
6. Section: LOW (compact table, 1 row per finding)
7. Section: PASSED CHECKS (checklist)
8. Section: Next steps (1-2 lines)
9. Footer (generated by, disclaimer, save location)
10. JSON summary (canonical EN, fenced ```json) — for machine consumption
```

**Verbose level by severity (v0.3+):**

| Severity | Format |
|---|---|
| CRITICAL | Overview table + full block (Short description + Why dangerous + Attacker scenario + Code before/after + Read more) |
| HIGH | Overview table + medium block (Description + Impact + Code fix + Read more) |
| MEDIUM | Compact table only (File:Line, Rule, Description) |
| LOW | Compact table only (File:Line, Rule, Description) |

Purpose: A non-tech user reading a CRITICAL finding should understand the impact and remediation steps without searching external resources.

### 🔴 [1] CRITICAL — SQL-INJECTION at `api/users.ts:42`

**{header_short_description}:** User-controlled input `req.body.id` concatenated into SQL query without parameterization.

**{header_why_dangerous}**

This is the number one vulnerability in web history. An attacker only needs to input a few characters into the URL/form to: dump all users/passwords, delete the database, or deploy backdoors to compromise the server. OWASP classifies SQL injection as A03 in the Top 10 (2021). Over the past 5 years, the largest data leaks (Equifax, LinkedIn, Yahoo) all involved SQL injection.

**{header_attack_scenario}**

1. Open browser and access: `https://yoursite.com/users?id=1' OR '1'='1`
2. Code runs as the SQL query: `SELECT * FROM users WHERE id = 1' OR '1'='1`
3. The condition `'1'='1'` is ALWAYS TRUE → DB returns **ALL users** (including admin)
4. Attacker sends `id=1 UNION SELECT password,email FROM users` → dumps all passwords
5. Exploitation time: 5-15 minutes to leak the entire database

**{header_current_code}**

\```javascript
const id = req.body.id  // {phrase_input_untrusted}
db.query(`SELECT * FROM users WHERE id = ${id}`)  // {phrase_concat_to_sink} SQL
\```

**{header_safe_code}**

\```javascript
const id = req.body.id
db.query('SELECT * FROM users WHERE id = ?', [id])  // ? is placeholder, DB engine safely parameterizes input
// Or using ORM (safe by default):
const user = await User.findOne({ where: { id } })
\```

**{header_read_more}:** [Rule detail](skills/scan-sec/rules/generic/02-sql-injection.md) · [OWASP A03 Injection](https://owasp.org/Top10/A03_2021-Injection/)

---

### 🔴 [2] CRITICAL — HARDCODED-SECRET at `.env:1`

**{header_short_description}:** Stripe live key committed to repository, anyone who clones or views source can see it.

**{header_why_dangerous}**

GitHub has bots scanning 24/7 for exposed secrets. When you `git push` a `.env` file, the key is detected within seconds. With a Stripe live key, attackers can charge cards or drain accounts; with an AWS key, they can compromise the entire infrastructure and cause massive financial billing.

**{header_attack_scenario}**

1. Repo pushed to public GitHub
2. Bots (TruffleHog, Gitleaks, GitGuardian, or custom attacker script) scan the new commit within seconds
3. Matches regex `sk_live_...` → extract key
4. Test key via Stripe API → confirm live status
5. Create unauthorized charges or refunds
6. Time from push to exploitation: **under 5 minutes**

**{header_current_code}**

\```bash
# .env (committed to git)
STRIPE_KEY=sk_live_EXAMPLE_REPLACE_WITH_REAL_KEY_IN_PROD
\```

**{header_safe_code}**

\```bash
# 1. ROTATE KEY IMMEDIATELY (assume it is compromised)
# 2. Remove .env from git history:
git rm --cached .env
echo ".env" >> .gitignore
git add .gitignore
git commit -m "Untrack .env"
# 3. Force rewrite history if repo was public:
# git filter-repo --path .env --invert-paths
# 4. Inject key via environment variables or secrets manager (AWS Secrets Manager, Vault, Doppler)
\```

**{header_read_more}:** [Rule detail](skills/scan-sec/rules/generic/01-hardcoded-secret.md)

### 🟠 [4] HIGH — WEAK-PASSWORD-HASHING at `auth.ts:18`

**{header_short_description}:** Passwords hashed using MD5 — cracked in seconds using GPUs/rainbow tables.

**{header_impact}**

If the database is leaked (via SQL injection, backup exposure, or internal breach), ALL user passwords can be decrypted quickly. Attackers can reuse these credentials across other sites (credential stuffing — how 80% of account takeovers occur in practice).

**{header_fix_code}**

\```javascript
// REPLACE:
const hash = crypto.createHash('md5').update(password).digest('hex')

// WITH bcrypt (industry standard):
const bcrypt = require('bcrypt')
const hash = await bcrypt.hash(password, 12)  // cost factor 12 = slow enough to block brute-force attempts

// OR argon2 (modern, recommended by OWASP):
const argon2 = require('argon2')
const hash = await argon2.hash(password)
\```

## Rendering Rules

### 1. Hide Sections with Zero Findings

If there are no findings in a section (e.g., no CRITICAL), omit the heading `## {section_critical}` and its table completely. Do not render empty tables.

### 2. Severity Capped by Rule

Each rule in `rules/generic/*.md` has a `severity_max` defined in its frontmatter. When raising a finding, the assigned severity must not exceed `severity_max`. You may downgrade the severity if the specific context represents lower risk.

Example: Rule `XSS` has `severity_max: HIGH`. If the vulnerability is found inside an admin-only page requiring multi-factor auth, you may downgrade it to MEDIUM. Note the downgrade reasoning in the Issue column.

### 3. File:Line Formatting

- Specify precise line numbers. If the rule applies to the whole file (e.g. committed `.env`): use `:1`.
- If the finding spans multiple lines: use `:start-end` (e.g. `routes.ts:42-58`).
- File paths must be relative to git root, using forward slashes `/` (even on Windows).

### 4. Concise Issue Descriptions

Keep descriptions to a maximum of 1 line, optionally including a short code snippet (~60 chars) in backticks. Technical snippets should remain untranslated.

### 5. Fix Descriptions

Prefer using i18n phrase templates (`{phrase_use_parameterized}`, `{phrase_use_bcrypt}`, etc.). If custom messaging is required, write it in the target language but keep technical terms (functions, library names) in English.

### 6. PASSED Section

List all rules checked where **no issues were detected**. Format:
```
- ✓ <Rule Name> — <1-line brief explanation of what was verified>
```

Rules that are not applicable (e.g. PHP rules checked in a Node repo) should be omitted from the PASSED list.

### 7. End JSON Summary

**ALWAYS use English keys** in the JSON block, regardless of `lang=vi` or `en`. This block is consumed by CI/CD scripts. Fields:

| Field | Type | Description |
|---|---|---|
| `verdict` | `"PASS"\|"WARN"\|"FAIL"` | Scan result summary |
| `summary.critical` | int | Count of **findings** (not rule hits) marked CRITICAL |
| `summary.high` | int | Count of findings marked HIGH |
| `summary.medium` | int | Count of findings marked MEDIUM |
| `summary.low` | int | Count of findings marked LOW |
| `summary.passed` | int | Count of rules checked and passed (max 21 in generic) |
| `scope` | string | Scope label (canonical EN) |
| `files_reviewed` | int | Number of files scanned |
| `primary_language` | string | lowercase, e.g. `"typescript"`, `"go"`, `"php"` |
| `specialized_rules_used` | bool | Whether language-specific overrides were applied |
| `mode` | `"small"\|"large"` | Execution mode |
| `date` | ISO 8601 date | Run date |
| `findings[]` | array | Array of objects containing: file, line, rule_id, severity, issue_summary, fix_summary |
| `top_rules_by_count[]` | array (optional) | Optional ranking; each entry contains: `{rule_id, count}` |

**Finding Count Protocol (CRITICAL to prevent math mismatches):**

1. **Each finding has exactly ONE primary `rule_id`.** Do not group multiple rule IDs into one entry.
2. **One line of code can trigger multiple findings if multiple rules are violated.** E.g., `routes/wallet.ts:12` has both IDOR and RACE-CONDITION → create 2 findings, each with its own rule ID.
3. **`summary.critical/high/medium/low` = number of entries in the `findings[]` array** matching that severity. The sum `critical + high + medium + low` must exactly equal `findings.length`.
4. **`top_rules_by_count[]` (if rendered) groups `findings[]` by `rule_id` and counts occurrences.** The sum of counts must exactly equal `findings.length` (no double-counting).
5. Cross-references in `issue` (e.g., "see also XSS at line X") are considered **notes** and do NOT create secondary findings.

**Sanity Check when Rendering:**
```
assert findings.length == summary.critical + summary.high + summary.medium + summary.low
assert findings.length == sum(top_rules_by_count[].count)  # if top rules are rendered
```

If the sanity check fails, re-check the aggregation step logic.

### 8. Required Footer Disclaimer

Every report must include the `{footer_disclaimer}` text — scan-sec is a reference scanner, NOT a replacement for manual professional security review.

## Edge cases

| Scenario | Rendering Rule |
|---|---|
| Repo passes completely (no findings) | Render header + verdict PASS + PASSED section + footer. Omit CRITICAL/HIGH/MEDIUM/LOW sections entirely. |
| Empty scope (no files) | Render header with "Files: 0" + message "{msg_no_files}". Omit findings sections. Verdict = PASS (default). |
| Multi-language repo | `header_primary_lang` displays combined, e.g. `"go + vue (both using specialized)"`. |
| Rule not applicable | Omit from PASSED list. E.g., PHP rules checked in a Node repo. |
| Sub-agent (LARGE mode) finding conflicts | Main agent dedups by `file:line:rule_id`, keeping the entry with the highest severity. |
